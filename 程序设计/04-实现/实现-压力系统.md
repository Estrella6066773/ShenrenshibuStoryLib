## 1. 职责边界

- **负责**：
  - `StressSystem` 作为玩家压力值的**统一变更入口**（`ApplyStressDelta`）。
  - 压力**分段（Tier）**解析与变化通知（本地 `event`/`Action` + 全局 `IEventMessage`）。
  - 进入高压/临界时的**压力事件**触发、活跃事件管理与**去重**（含满值性格失控独立去重表）。
  - 对委托范围内的**任务 Scoped** 临时效果（效率/结算通道乘数，按 `assignmentId` 与过期日存活）。
  - 与**邮件**、**员工状态**的桥接（不直接操作邮件 UI）。
- **不负责**：矛盾案件状态机与协调流程（`ConflictSystem`）；邮件/聊天 UI 渲染（`MessageSystem`）。

## 2. 场景入口与生命周期

典型挂载（可与其它系统根同级或子级）：

| 组件 | 说明 |
|------|------|
| `StressSystem` | 核心逻辑；Inspector 绑定 `StressConfigSO`、`StressEventsCatalogSO`，可选 `PlayerEconomySystem` |
| `StressMailBridge` | 订阅压力事件消息 → `MailDispatchService.TrySendByEventId` |
| `StressEmployeeStateBridge` | 订阅 `StressSystem` 本地事件 → 员工 `InStressEvent` 与恢复 |

生命周期要点：

- `Awake`：校验配置阈值、钳制初始压力、同步经济系统日/时间槽。
- `OnEnable`：订阅 `PlayerEconomySystem.OnDayChanged`（日终恢复、Scoped 效果刷新、持续临界检查等）。
- 所有压力增量应经 `ApplyStressDelta`；内部发布 `StressValueChangedEvent` / `StressTierChangedEvent`，并在条件满足时触发压力事件。

## 4. 核心行为摘要

### 4.1 压力变更与分段

- 入口：`ApplyStressDelta(string source, int delta, string employeeId = "", bool forceEventTrigger = false)`。
- 对外只读：`StressValue`、`CurrentTier`（由当前值与 `StressConfigSO` 阈值解析）。
- 变更后：
  - 触发 `OnStressChanged`；
  - 发布 `StressValueChangedEvent`；
  - 若 Tier 变化：触发 `OnStressTierChanged`，发布 `StressTierChangedEvent`。

### 4.2 压力事件与去重

- 常规事件：使用 `_dedupeKeys`（`HashSet<string>`），同一 `dedupeKey` 仅触发一次。
- 满值性格失控：使用独立 `_meltdownDedupeKeys`，键形如 `meltdown|{defKey}|{day}|{timeSlot}`。
- 触发路径（概念）：
  1. 压力达到上限时优先尝试 `TryResolvePressureMaxEvent`（按玩家性格标签匹配 `StressEventsCatalogSO`）；
  2. 否则在**跨入高压/临界**或 `forceEventTrigger` 时走常规模板触发。
- 失控类后果可包含：玩家异常标签、邮件投递（`hideMailChoices`）、任务 Scoped 降压/效率修正等（见 `StressPressureEventDefinitionSO` 与 `StressSystem` 内注释）。
- 活跃事件保存在 `_activeEventById`；结案经 `ResolveStressEvent` 等路径，发布 `StressEventResolvedEvent`。

### 4.3 任务 Scoped 效果

- 内部结构：`TaskScopedEffectEntry`（`assignmentId`、`expireDayExclusive`、按员工的效率/结算乘数表）。
- 随游戏日 `RefreshTaskScopedEffects` 剔除过期项；与委托 `assignmentId` 绑定，**不负责**委托生命周期本身。

### 4.4 员工与邮件桥接

- **员工**（`StressEmployeeStateBridge`）：
  - 事件创建：将涉事员工标为 `EmployeeStatus.InStressEvent`，`extraData` 记录原状态与岗位；
  - `skipsEmployeeStressBridge == true` 的事件（如部分满值失控）跳过员工状态；
  - 结案：按 `stressEventId` 扫描恢复，并发布 `StressEmployeeStateChangedEvent`。
- **邮件**（`StressMailBridge`）：
  - `StressEventTriggeredEvent` → `MailDispatchService.TrySendByEventId(mailEventId, mailId, dedupeKey)`；
  - `StressEventResolvedEvent` → 优先投递结案邮件 SO，否则在 `MailRuntimeInbox` 追加处理结果段落。

## 6. 消息与集成

### 6.1 本地回调（`StressSystem`）

- `OnStressChanged`、`OnStressTierChanged`
- `OnStressEventCreated`、`OnStressEventMailCreated`、`OnStressEventResolved`

### 6.2 全局消息（`StressMessages.cs` → `RuntimeMessageHub`）

| 消息 | 用途 |
|------|------|
| `StressValueChangedEvent` | 任意压力数值变化 |
| `StressTierChangedEvent` | 分段变化 |
| `StressEventTriggeredEvent` | 压力事件触发（含 `mailId`、`mailEventId`、`dedupeKey`、`hideMailChoices`、`isPersonalityMeltdown` 等） |
| `StressEventResolvedEvent` | 玩家选项结案 |
| `StressEmployeeStateChangedEvent` | 员工因压力事件进出特殊状态 |

`StressSystem` 通过类内 `PublishMessage<T>` 辅助方法转发到总线（需 `RuntimeMessageHub.IsReady`）。

### 6.3 关联模块

| 模块 | 耦合方式 |
|------|----------|
| `PlayerEconomySystem` | 序列化引用：当前日/时间槽、压力增量经济缩放、日终恢复 |
| `Employee` | `StressEmployeeStateBridge` + `EmployeeRuntimeDataRegistry` |
| `Assignment` | 任务 Scoped 效果按 `assignmentId`；`StressTaskPositionBanRegistry` 限制回槽 |
| `MessageSystem` | `StressMailBridge` + `MailDispatchService` / `MailRuntimeInbox` |

## 7. 存档

- 类型：`StressSaveData`（`stressValue`、Tier、活跃事件列表、去重相关字段等）。
- 由 `StressSystem` 提供导出/导入 API（与总存档对接时由上层聚合调用）。
