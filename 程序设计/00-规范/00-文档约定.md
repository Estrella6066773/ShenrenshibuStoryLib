# 文档与追溯约定

## 1. 命名与路径（统一使用连字符 `-`）

与 [机魂](D:\Unity\机魂) `Docs/` 一致：**编号目录、文档文件名中的分段符一律用 `-`**，不用 `_`。

| 类型 | 路径模式（本仓库根） | 示例 |
|------|----------------------|------|
| 需求 | `系统设计/02-需求/需求-<模块>.md` | `需求-压力系统.md` |
| 计划 | `系统设计/03-计划/计划-<主题>.md` | `计划-模块依赖与建议顺序.md` |
| 实现 | `程序设计/04-实现/实现-<模块>.md` | `实现-压力系统.md` |
| 交付 | `程序设计/05-交付/交付-<名称>.md` | `交付-追溯矩阵.md` |
| 产品 | `系统设计/01-产品/` | `产品愿景与边界.md` |

在 Unity 主仓中，上述路径位于 `Docs/ShenrenshibuStoryLib/` 下。

**例外（保留 `_`）：**

- `Docs/ShenrenshibuStoryLib/_templates/` — 模板库目录（与机魂相同）
- 需求 ID：`REQ-<MODULE>-<序号>`（本就用连字符）
- 正文层命名：遵循 [正文/治理/01-命名文件与ID规则](../../正文/治理/01-命名文件与ID规则.md)

**禁止：** `02-需求`、`需求_压力系统.md` 等混用 `_` 的目录或工程文档名。

## 2. 需求 ID（可选但推荐）

- 格式：`REQ-<MODULE>-<三位序号>`，例如 `REQ-ASG-001`。
- 同一需求改写时**保留 ID**，在变更记录中说明版本。

## 3. 状态字段（矩阵与 REQ 表头可共用）

| 状态 | 含义 |
|------|------|
| draft | 草稿，未对齐程序 |
| agreed | 策划/程序已共识 |
| implemented | 主路径已在代码中落地 |
| deferred | 明确推迟 |
| superseded | 被新 ID 取代 |

## 4. 实现文档（IMPL）维护规则

- **随代码变更更新**：合并影响行为的 PR 时，同一提交或紧随其后的提交更新对应 `实现-*` 文档。
- 类名、路径以仓库为准；长流程引用 [核心系统与核心循环](../01-架构总览/核心系统与核心循环.md) 中的 mermaid，避免 IMPL 与总架构图矛盾。

## 5. Unity 资源

- 工程文档在本仓库（子模块 `Docs/ShenrenshibuStoryLib/`），**不由 Unity 导入**。Unity 主仓 `Assets/00_Docs/` 仅保留跳转。

## 6. 策划与程序分工（建议）

| 文档 | 主责 | 协作者 |
|------|------|--------|
| REQ | 主策 | 程序（可行性）、测试（可测性） |
| PLAN | PM 或主程 | 主策 |
| IMPL | 程序 | 主策（核对玩家可见行为） |
| UAT / 矩阵 | 测试 / 主策 | 程序（自动化与调试入口） |
